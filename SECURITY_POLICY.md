# Security Policy

## Supported Versions

| Version | Status | Security Support |
| --------- | -------- | ----------------- |
| Current release (main branch) | Active development | Full security patches |
| Previous minor release | Maintenance | Critical security patches only |
| Older releases | End of life | No security support |

Security patches are applied to the current release branch. Users should
always run the latest version.

---

## Reporting a Vulnerability

**Do NOT report security vulnerabilities through public GitHub issues.**

### How to Report

1. **Email**: Send a detailed report to the project maintainers via private email
2. **Include**: Description, steps to reproduce, impact assessment, and any
   proof-of-concept code
3. **Encrypt**: Use GPG encryption if available (public key in repository)

### What to Include

- Type of vulnerability (XSS, CSRF, SQL injection, auth bypass, etc.)
- Full path to the affected source file(s)
- Step-by-step instructions to reproduce
- Proof-of-concept or exploit code (if applicable)
- Impact assessment (what data/functionality is at risk)
- Suggested fix (if you have one)

### What NOT to Do

- Do not publicly disclose before a patch is available
- Do not access or modify other users' data
- Do not perform denial-of-service testing on production
- Do not use automated scanning tools against production without permission

## Security Practices

### OWASP Top 10 Compliance

This project actively defends against the OWASP Top 10 (2021):

| OWASP Category | Defense Measures |
| ---------------- | ----------------- |
| A01: Broken Access Control | `@login_required` on protected views, `is_staff` checks on admin, ownership verification on object access, RBAC via Django permissions |
| A02: Cryptographic Failures | Secrets via environment variables, HTTPS enforced, secure cookie flags, Argon2 password hashing |
| A03: Injection | Django ORM exclusively (no raw SQL), nh3 HTML sanitization, parameterized queries enforced |
| A04: Insecure Design | Service layer architecture, `@transaction.atomic` on financial ops, `select_for_update()` on wallet mutations |
| A05: Security Misconfiguration | Production checklist enforced, `DEBUG=False`, HSTS headers, CSP headers, `X_FRAME_OPTIONS=DENY` |
| A06: Vulnerable Components | `requirements.txt` pinned versions, `pip-audit` checks, dependency monitoring |
| A07: Auth Failures | JWT authentication, MFA support, rate limiting on login endpoints, session timeout |
| A08: Data Integrity | Migration safety hooks, backup verification, signed download tokens (HMAC) |
| A09: Logging Failures | Structured logging, no PII in logs, audit trail via `AuditLog` model |
| A10: SSRF | File upload validation (MIME, extension, size), storage abstraction layer |

### HTML Sanitization

All user-supplied HTML content is sanitized using **nh3** (Rust-based,
memory-safe). The `bleach` library has been fully replaced.

```python
from apps.core.sanitizers import sanitize_html_content

clean_html = sanitize_html_content(user_input)
```

The sanitizer allows a controlled allowlist of HTML tags and attributes.
All JavaScript, event handlers, and dangerous attributes are stripped.

### Database Security

- **No raw SQL**: Django ORM is the only supported query interface
- **Parameterized queries**: Enforced by the ORM â€” never format user data
  into query strings
- **Transaction safety**: `@transaction.atomic` on all multi-table operations
- **Financial safety**: `select_for_update()` on all wallet/credit mutations
- **Connection pooling**: PostgreSQL connection pooling configured

### CSRF Protection

- All mutating endpoints (POST, PUT, PATCH, DELETE) require CSRF tokens
- HTMX requests include CSRF token via global `<body hx-headers>` attribute
- Django's `CsrfViewMiddleware` is always enabled
- `csrf_exempt` is forbidden without explicit security review

### Content Security Policy (CSP)

- CSP headers configured with nonce-based script authorization
- Per-request nonce generated by `csp_nonce` middleware
- `script-src` uses `nonce-{value}`, not `unsafe-inline`
- CSP violation reports collected for monitoring
- Inline scripts require the request-specific nonce attribute

### Rate Limiting

This platform has **two separate rate limiting systems**:

| System | Location | Purpose |
| -------- | ---------- | --------- |
| WAF Rate Limiting | `apps.security` (`RateLimitRule`, `BlockedIP`, `CrawlerRule`) | IP/path-based DDoS and bot protection at middleware level |
| Download Quotas | `apps.firmwares` (`DownloadToken`) + `apps.devices` (`QuotaTier`) | Per-user/tier download limits at firmware download time |

These systems are independent and must never be conflated.

### Input Validation

- All form inputs validated through Django forms or DRF serializers
- File uploads verified: MIME type, file extension, and file size
- API inputs validated through DRF serializer fields
- URL parameters validated before database queries
- No trust of client-side validation â€” server always re-validates

### Secrets Management

- All secrets loaded from environment variables
- `.env` files are gitignored and never committed
- Service account credentials stored in `storage_credentials/` (gitignored)
- No secrets, API keys, or passwords in source code
- Git history periodically scanned for accidental secret commits
- Secrets rotation procedures documented

## Scope

### In Scope

- Web application (Django views, templates, static files)
- REST API (DRF endpoints, authentication, authorization)
- Authentication system (login, registration, JWT, MFA, social auth)
- Download pipeline (token generation, ad gate, file serving)
- Payment/wallet system (credit management, transactions)
- Admin panel (staff authentication, data management)
- Background tasks (Celery workers, scheduled jobs)
- File upload handling (firmware files, user avatars, attachments)

### Out of Scope

- Hosting infrastructure (server configuration, OS hardening)
- Third-party services (CDN, email providers, payment gateways)
- Client-side browser vulnerabilities
- Physical security
- Social engineering attacks against team members
